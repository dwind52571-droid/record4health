# 健康记录器 MVP 方案

## 1. 产品目标

做一个适合手机使用的轻量记录工具，帮助用户低摩擦地记录：

- 每日体重
- 每日饮食
- 每日运动
- 每周 / 每月统计

核心目标不是做一个“大而全”的健身 App，而是优先保证：

- 记录快
- 手机上方便打开和操作
- 数据看得清楚
- 先尽快上线一个自己愿意持续使用的版本

## 2. 目标用户

当前目标用户默认只有 1 类：产品创建者本人。

这意味着第一版可以刻意简化：

- 不做多人系统
- 不做复杂社交
- 不做专业级营养分析
- 不做复杂权限体系

## 3. 产品形式

第一版推荐采用：

- 手机端 Web App
- 支持 PWA，方便添加到手机桌面
- 移动端优先设计

选择这个形式的原因：

- 开发速度快
- 成本低
- 迭代灵活
- 后续可以再封装成 App 或小程序

## 4. 核心使用流程

### 早上

- 用户打开首页
- 记录空腹体重
- 可选填写备注

### 三餐

- 用户拍照上传早餐 / 午餐 / 晚餐
- AI 识别食物内容
- 系统返回食物项、估算分量、估算热量
- 用户确认或手动修改后保存

### 晚上运动

- 用户选择运动类型
- 输入时长或次数
- 系统根据体重和运动类型估算热量消耗

### 查看统计

- 查看今日摄入、消耗、净值
- 查看每周 / 每月的体重趋势和行为统计

## 5. MVP 功能范围

### 必做

- 用户基础信息
- 体重记录
- 饮食记录
- 食物照片上传
- AI 识别食物并估算热量
- 运动记录
- 运动热量估算
- 今日汇总页
- 每周统计
- 每月统计

### 可做但不阻塞第一版

- 手动补录历史数据
- 饮水记录
- 目标体重设置
- 常用食物快捷添加
- 常用运动快捷添加

### 暂不做

- 社交功能
- 排行榜
- 教练体系
- 穿戴设备接入
- 极其精细的宏量营养素数据库
- 自动生成完整饮食方案

## 6. 页面结构

### 今日页

建议作为默认首页，展示：

- 今日体重
- 今日总摄入热量
- 今日总消耗热量
- 今日净热量
- 今日早餐 / 午餐 / 晚餐记录
- 今日运动记录
- 快速记录入口

### 记录页

用于快速新增数据：

- 记录体重
- 拍照记饮食
- 手动添加饮食
- 记录运动

### 统计页

展示可量化趋势：

- 近 7 天体重折线图
- 近 30 天体重折线图
- 每周摄入 / 消耗汇总
- 每月摄入 / 消耗汇总
- 记录完成率

### 我的页

- 身高
- 性别
- 年龄
- 当前体重目标
- 常用运动设置
- 常用食物偏好

## 7. 关键交互原则

### 原则 1：3 秒内开始记录

首页必须直接提供入口，不让用户层层跳转。

### 原则 2：AI 先给结果，用户只做确认

拍照后不要求用户手工填写所有字段，系统应该先识别并给建议。

### 原则 3：允许不精确，但要持续可用

第一版热量估算可以接受“大致准确”，重点是帮助形成长期记录习惯。

### 原则 4：统计比单条记录更重要

用户最终关心的是趋势，而不只是某一顿饭的数字。

## 8. 指标定义

### 日维度

- 当日体重
- 当日摄入热量
- 当日运动消耗热量
- 当日净热量
- 当日记录次数

### 周维度

- 周平均体重
- 周体重变化
- 周总摄入热量
- 周总消耗热量
- 周记录天数

### 月维度

- 月平均体重
- 月体重变化
- 月总摄入热量
- 月总消耗热量
- 月记录完成率

## 9. 推荐技术路线

### 前端

- Next.js
- TypeScript
- Tailwind CSS
- PWA 支持

### 后端 / 数据

- Supabase
- PostgreSQL
- Supabase Storage 用于图片上传

### AI 能力

- 多模态模型识别食物图片
- 文本模型结构化输出食物项和热量估算

## 10. 下一步拆解

建议按下面顺序推进：

1. 确认 MVP 页面和字段
2. 设计数据库表
3. 搭建 Web 项目
4. 先做手动记录流程
5. 再接入 AI 识图
6. 最后补统计图表和 PWA 安装体验
